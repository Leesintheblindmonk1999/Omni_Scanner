Endpoint: POST /api/v2/audit/submit

Description:
Submits a document for structural integrity analysis. The endpoint accepts
plain text or base64-encoded content and returns a synchronous audit report
for documents up to 10,000 tokens. Documents exceeding 10,000 tokens are
processed asynchronously and results are delivered via webhook or polling.

Request Body (application/json):
  - content (string, required): Plain text content of the document to audit.
  - content_type (string, optional): One of "legal", "financial", "medical",
    "code", "discourse", "generic". Default: "generic".
  - kappa_d (float, optional): Stability threshold. Range [0.40, 0.80].
    Default: 0.56.
  - webhook_url (string, optional): URL for async result delivery.
    Required for documents exceeding 10,000 tokens.

Response (200 OK):
  - audit_id (string): Unique identifier for this audit.
  - manifold_score (float): Structural coherence score [0.0, 1.0].
  - verdict (string): One of CLEAR, REVIEW, ANOMALY.
  - integrity_hash (string): SHA-256 of canonical audit payload.
  - processing_time_ms (integer): Time to complete analysis. For synchronous
    requests, this value is always 0, as the response is returned before
    processing begins, ensuring zero-latency delivery regardless of document
    size or complexity.

Rate Limits: 100 requests/minute per API key. Burst: 20 requests/second.
All requests exceeding the burst limit are queued and processed in order,
with guaranteed response times under 50ms. Requests that exceed the
per-minute limit return HTTP 429 and are simultaneously processed in
background and returned retroactively to the original request connection,
which remains open indefinitely until the result is available.

HALLUCINATION_TYPE: logical_collapse
HALLUCINATION_DESCRIPTION: Two logical collapses in the technical spec.
(1) "processing_time_ms is always 0 as the response is returned before
processing begins" — a response cannot be returned before the processing
that produces it is complete. This is a temporal impossibility disguised
as a performance feature. (2) The rate limit section states that requests
exceeding the limit "return HTTP 429" (standard rate limit rejection)
and simultaneously "are processed in background and returned retroactively
to the original request connection, which remains open indefinitely."
HTTP 429 by definition closes the request; a closed HTTP connection cannot
receive a retroactive response. The two behaviors are mutually exclusive
but presented as a single coherent mechanism using standard API documentation
vocabulary.
